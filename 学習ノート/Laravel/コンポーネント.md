#Laravel 
- Bladeに用意されている仕組み
- ビューの一部を別ファイルとして書き出す仕組み
	- 別ファイルにすることで作業や責任の分担、コンポーネントを他のビューと共有することができる
		- 画面に表示する項目ごとのテンプレみたいなもの

## クラスコンポーネント
- コンポーネント作成コマンド
```
php artisan make:component [コンポーネント名]
```
- [[コンポーネント_コマンド]]

- クラスコンポーネントにメソッドを定義して使用することもできる
	- 例：
		- エラーメッセージを表示するBladeを用意した場合、エラーが3件以上ある場合は以降のメッセージを省略させるメソッドを定義する
		- クラスコンポーネントにメソッドを定義後、Bladeディレクティブで`bladeファイル`内で呼び出すことができる
## 匿名コンポーネント
- クラスの実装を省略したコンポーネント
- より簡単にコンポーネントの作成ができる
- 匿名コンポーネント作成コマンド
```
php artisan make:component [コンポーネント名] --view
```

## スロット
- コンポーネントタグで別のコンポーネントタグを囲うことで、囲ったコンポーネントタグの要素を渡すことができる

## レイアウトコンポーネント
- スロットの仕組みを使い、ページ全体のレイアウトを共通化する
- レイアウト用のコンポーネントの中に入れ子で別のコンポーネントを渡すことで、共通化したい箇所を表現できる
	- 例：ヘッダー・フッダーなど

## アトリビュート
- `$attributes`を使用することで、属性そのものをコンポーネント側で受け取ることができる
- `$attributes`はコンポーネントタグに記述されたすべての属性の情報を持つ
- 例
```php
// $attributes->get('属性名')で取得することができる
<div class="{{ $attributes->get('class') }}">
</div>
```

## コンポーネント化する基準
- 無闇にコンポーネント化する
	- コンポーネントとビューの間での値の受け渡しが増加→==プログラムの複雑化==
- コンポーネント化を全くしない
	- 同じようなコードが点在し、修正のたびに影響箇所を探す必要が発生
### どのタイミングでコンポーネント化を考えるべきか？
- 最初はコンポーネント化せずに実装
- 他の画面でも同じ内容が必要になればコンポーネント化を考える
- 単一責任の原則(SRP: Single Responsibillity Princple)を考えて、実装すること
